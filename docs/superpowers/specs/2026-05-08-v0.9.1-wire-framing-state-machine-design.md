# UnrealClaudeMCP v0.9.1 — Wire-framing partial-message state machine

**Date:** 2026-05-08
**Status:** Approved (autonomous-mode design — user-authorized "do everything" execution; spec stands as the design contract)
**Author:** Claude (with Najem)
**Target release:** v0.9.1
**Builds on:** v0.9.0 (32 handlers; latest main)
**Scope:** Refactor `FUCMCPServer::TickClients` and the framing helpers in `MCPServer.cpp` so that the wire protocol correctly distinguishes "no data available right now" (would-block on a non-blocking socket) from "connection closed," and resumes partial frame transfers across `TickClients` invocations.

---

## Goal

Make the bridge↔plugin TCP wire protocol robust under TCP fragmentation. Right now a single frame whose bytes happen to arrive in chunks across multiple 50ms ticker invocations gets mistaken for a closed connection, and the client is dropped mid-message. On localhost loopback the bug rarely fires (the kernel typically delivers whole frames in one packet), but it surfaces predictably for large messages — most concretely a 2 MB+ viewport screenshot — where the call appears to hang and the caller has to retry.

After this change, frames of arbitrary size assemble correctly across as many tick boundaries as needed, with no spurious disconnects.

Tool count unchanged (still 32). Version: 0.9.0 → 0.9.1 (semver patch bump — protocol robustness fix, no new capability, no API surface change).

---

## Scope

### In scope

- Replace the all-or-nothing `ReadFramedMessage` / `WriteFramedMessage` helpers with non-blocking, resumable variants.
- Add per-client read/write state structs (`FUCMCPClientReadState`, `FUCMCPClientWriteState`) tracked by `FUCMCPServer`.
- Disambiguate would-block (`SE_EWOULDBLOCK`) from connection-closed via `ISocketSubsystem::GetLastErrorCode()`.
- `TickClients` refactored to drive both state machines; cleanup of state on disconnect / server stop.
- Pipelined frames: a single tick should drain as many complete frames as possible before yielding (so a backlogged client catches up quickly).
- Smoke test addition: an explicit "large response" round-trip (fire `get_viewport_screenshot` → expect a successful base64 PNG > 100 KB) so the regression is visible if the state machine breaks.

### Out of scope

- Wire protocol format change. Still 8-byte big-endian length prefix + UTF-8 body. The bridge side (Python) is untouched.
- TLS / authentication. Localhost-only assumption stands.
- Multiple frames in flight concurrently per client (the protocol is request/response sync; one frame at a time is correct).
- Backpressure / flow control. UE's socket buffer is the only buffering layer; if it fills, sends fail and the client is dropped (same as today).
- Frame size limit beyond the existing 1 GB cap.
- IPv6 / external interface listening.

## Architecture

### Per-client state structs

```cpp
// In MCPServer.h (or a new MCPServerInternal.h)
struct FUCMCPClientReadState
{
    // Phase 1: 8-byte length prefix
    uint8 PrefixBuf[8] = {};
    int32 PrefixAccum = 0;

    // Phase 2: body
    uint64 BodyLength = 0;          // decoded once PrefixAccum hits 8
    TArray<uint8> BodyBuf;          // sized once BodyLength is known
    int32 BodyAccum = 0;

    // Helpers
    bool IsHeaderComplete() const { return PrefixAccum >= 8; }
    bool IsBodyComplete() const { return IsHeaderComplete() && BodyAccum >= static_cast<int32>(BodyLength); }
    void Reset() { *this = FUCMCPClientReadState(); }
};

struct FUCMCPClientWriteState
{
    TArray<uint8> PendingBytes;     // 8-byte prefix + UTF-8 body, concatenated
    int32 BytesSent = 0;
    bool IsDrained() const { return BytesSent >= PendingBytes.Num(); }
    void Reset() { PendingBytes.Reset(); BytesSent = 0; }
};
```

Each client gets one of each, lifetime-managed by `FUCMCPServer` via two TMaps keyed by `FSocket*`. State is created on `OnConnectionAccepted`, destroyed on disconnect / server stop.

### Result enum for non-blocking helpers

```cpp
enum class EUCMCPFrameResult : uint8
{
    Complete,    // A whole frame is now available; consume it
    InProgress,  // Made progress (or none) — try again next tick
    Disconnect,  // Real error; drop the client
};
```

### Refactored helpers

`AdvanceRead(FSocket*, FUCMCPClientReadState&) -> EUCMCPFrameResult`
- Phase 1: while `PrefixAccum < 8`, call `Recv`. Return `InProgress` on would-block, `Disconnect` on error, advance `PrefixAccum` on data.
- Phase 1→2 transition: when `PrefixAccum == 8` and `BodyLength` is still 0, decode the prefix; validate (`BodyLength != 0`, `BodyLength <= 1 GB`); allocate `BodyBuf`.
- Phase 2: same loop pattern on the body bytes.
- Returns `Complete` only when `IsBodyComplete()`.

`AdvanceWrite(FSocket*, FUCMCPClientWriteState&) -> EUCMCPFrameResult`
- Loop `Send` against `PendingBytes + BytesSent`, advance on success, return `InProgress` on would-block, `Disconnect` on error.
- Returns `Complete` once `IsDrained()`.

### Would-block disambiguation

Per UE 5.7 source (`SocketsBSD.cpp` — `bStreamSocket && (TranslateErrorCode(BytesRead) == SE_EWOULDBLOCK)` is treated as success-with-no-data), the canonical disambiguation is:

```cpp
// After a Recv that returned bOk == false OR bOk == true with BytesRead == 0:
ISocketSubsystem* Subsys = ISocketSubsystem::Get(PLATFORM_SOCKETSUBSYSTEM);
const ESocketErrors LastErr = Subsys ? Subsys->GetLastErrorCode() : ESocketErrors::SE_NO_ERROR;
if (LastErr == ESocketErrors::SE_EWOULDBLOCK)
{
    // Try again next tick
    return EUCMCPFrameResult::InProgress;
}
// Real error — could be SE_ECONNRESET, SE_ENOTCONN, etc.
return EUCMCPFrameResult::Disconnect;
```

This same check applies on Send-side.

### TickClients refactor

```cpp
bool FUCMCPServer::TickClients(float /*DeltaTime*/)
{
    TArray<FSocket*> Dropped;
    for (FSocket* Sock : ConnectedClients)
    {
        if (!Sock || Sock->GetConnectionState() != SCS_Connected)
        {
            Dropped.Add(Sock);
            continue;
        }

        // 1) Drain any pending write state first. If the client has a partial
        //    response queued, push it before reading more requests.
        FUCMCPClientWriteState& W = WriteStates.FindOrAdd(Sock);
        if (!W.IsDrained())
        {
            const EUCMCPFrameResult WR = AdvanceWrite(Sock, W);
            if (WR == EUCMCPFrameResult::Disconnect) { Dropped.Add(Sock); continue; }
            if (WR == EUCMCPFrameResult::InProgress) { /* still flushing — read can wait */ continue; }
            // WR == Complete: write done, can now read more
            W.Reset();
        }

        // 2) Drain as many complete read frames as are available this tick.
        FUCMCPClientReadState& R = ReadStates.FindOrAdd(Sock);
        bool bDropAfter = false;
        for (int32 SafetyBound = 0; SafetyBound < 32; ++SafetyBound)
        {
            const EUCMCPFrameResult RR = AdvanceRead(Sock, R);
            if (RR == EUCMCPFrameResult::Disconnect) { bDropAfter = true; break; }
            if (RR == EUCMCPFrameResult::InProgress) { break; }
            // RR == Complete: dispatch and queue the response
            const FString Body = ConsumeReadBody(R);  // converts BodyBuf -> FString, resets state
            const FString Resp = FUCMCPDispatcher::HandleMessage(Body);
            if (!Resp.IsEmpty())
            {
                EncodeFrame(Resp, W.PendingBytes);  // builds 8-byte prefix + UTF-8 body
                W.BytesSent = 0;
                // First write attempt this tick; if it doesn't drain, we'll resume next tick
                const EUCMCPFrameResult WR2 = AdvanceWrite(Sock, W);
                if (WR2 == EUCMCPFrameResult::Disconnect) { bDropAfter = true; break; }
                if (WR2 == EUCMCPFrameResult::InProgress) { break; }
                W.Reset();
            }
        }
        if (bDropAfter) { Dropped.Add(Sock); }
    }

    // 3) Cleanup dropped sockets and their state.
    ISocketSubsystem* Subsys = ISocketSubsystem::Get(PLATFORM_SOCKETSUBSYSTEM);
    for (FSocket* Sock : Dropped)
    {
        ConnectedClients.Remove(Sock);
        ReadStates.Remove(Sock);
        WriteStates.Remove(Sock);
        if (Sock && Subsys)
        {
            Sock->Close();
            Subsys->DestroySocket(Sock);
        }
    }
    return true;
}
```

The `SafetyBound` of 32 frames per tick is a fairness guard — it prevents one busy client from monopolizing the tick. With a 50ms ticker, 32 small frames per client per tick is generous.

## Risk surface and mitigations

| Risk | Mitigation |
|---|---|
| `TMap<FSocket*, ...>` lifetime: if a client drops and we forget to remove its entry, memory leaks across clients | Cleanup paths in three places: (1) the `Dropped` loop in `TickClients`; (2) the `Stop()` method's iteration over `ConnectedClients`; (3) the destructor (which calls `Stop()`). |
| `FindOrAdd` semantics: the first call for a new socket creates a default state — fine, but if `OnConnectionAccepted` is called after the state is already created, it'd be silently overwritten. | `OnConnectionAccepted` should call `ReadStates.Add(InSocket, FUCMCPClientReadState{})` and `WriteStates.Add(InSocket, FUCMCPClientWriteState{})` explicitly so the lifecycle is visible at insertion. |
| Pipelined frames: 32-frame safety bound could starve other clients if one client floods | The bound is per-tick-per-client; fair scheduling across clients comes from the outer `for (FSocket* Sock : ConnectedClients)` loop. |
| Frame body of 0 length still treated as `framing_error` | Preserved from v0.5.0. The 0-length check moves into `AdvanceRead` after the prefix is fully decoded. |
| `GetLastErrorCode()` return value can be stale across threads | All socket I/O happens on the game thread (FTSTicker), so single-threaded. No race. |
| Existing in-flight frames at moment of upgrade | Server-side restart on plugin reload is required anyway (the plugin is `Type: Editor`), so no live-upgrade concern. |

## Test strategy

**No new pytest unit tests.** The bridge's pytest suite covers the Python side; the wire-framing fix is pure C++. Mocking `FSocket` to test partial reads in isolation would require interface refactoring that's out of scope.

**Integration coverage** via the smoke test:
- Existing smoke steps continue to pass — confirms the refactor didn't regress small-frame round-trips.
- New `large_response` step fires `get_viewport_screenshot` and asserts the result is a valid base64 PNG of at least 100 KB. This is the concrete scenario where the v0.5.0 bug was most likely to fire (large bodies → fragmentation → spurious disconnect).

**Manual verification** — open two MCP clients simultaneously, fire requests on both, ensure neither gets dropped due to the other's traffic. Document this as a one-time check, not a recurring CI gate.

## What this fixes (concretely, in human terms)

Before: Claude asks for a viewport screenshot (~2 MB PNG). The plugin's response is split into multiple TCP packets by the kernel. The first packet arrives within the current tick; the rest arrive on the next tick. The plugin's read loop sees "no more data right now" between those packets, mistakes it for "client disconnected," drops the connection. Claude gets a connection error, retries, usually succeeds. Annoying.

After: Same scenario. The plugin reads what's available, saves the partial buffer, returns from the tick. Next tick (50ms later), reads more, completes the frame, dispatches, sends the response. No drop. No retry needed.
