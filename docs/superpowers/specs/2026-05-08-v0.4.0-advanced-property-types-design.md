# UnrealClaudeMCP v0.4.0 — Advanced Property Types design

**Date:** 2026-05-08
**Status:** Approved — proceeding to implementation plan
**Author:** Claude (with Najem)
**Target release:** v0.4.0
**Builds on:** v0.3.0 (build-a-level — 6 handlers + ActorIdentity + PropertyCoercion shared modules)
**Closes:** the explicit "USTRUCT / TArray / TMap / TSet / FInstancedStruct deferred to v0.4.0" deferral from v0.3.0's spec.

---

## Goal

Extend `PropertyCoercion` (the JSON↔FProperty bridge added in v0.3.0) to support **complex / nested UE property types**. **No new MCP handlers.** The three handlers that already use `PropertyCoercion` — `spawn_actor.properties`, `set_actor_property`, `add_component.relative_transform` — all benefit transparently.

This is a deliberate "code-only" release: tool count stays at **19**. The win is in capability depth, not surface breadth.

## Scope

### In scope (this PR)

#### Type-level coverage

| Type | JSON encoding | Example |
|---|---|---|
| **USTRUCT** (any reflected struct) | JSON object → field-by-field via recursive coercion | `{"X": 100, "Y": 50, "Z": 0}` → `FVector` |
| **TArray<T>** | JSON array → element-wise | `[1, 2, 3]` → `TArray<int32>` |
| **TMap<FString, V>** | JSON object | `{"foo": 1, "bar": 2}` → `TMap<FString, int32>` |
| **TMap<FName, V>** | JSON object | `{"foo": 1}` → `TMap<FName, int32>` |
| **TSet<T>** | JSON array (deduplicated server-side) | `[1, 2, 3]` → `TSet<int32>` |
| **FObjectProperty** (hard `UObject*` pointer) | JSON string (asset path) → `LoadObject` + class check | `"/Game/Meshes/SM_Cube.SM_Cube"` → `UStaticMesh*` |

#### Property-name path traversal (NEW)

The `property` param of `set_actor_property` (and the keys in `spawn_actor.properties`) accept **dotted path syntax** to traverse through nested objects and structs:

| Path | Resolves to |
|---|---|
| `Tags` | Top-level UPROPERTY on the actor |
| `RootComponent.RelativeLocation` | RootComponent (FObjectProperty → dereference) → RelativeLocation (FStructProperty `FVector`) |
| `RootComponent.RelativeRotation.Yaw` | Three hops: Object → Struct → primitive field |
| `PrimaryActorTick.bCanEverTick` | Two hops: Struct → primitive bool |

Without path traversal, USTRUCT/FObjectProperty support is half-useful — most interesting properties (transforms, light parameters, mesh references) live on components, not on the actor's top-level properties. Path traversal is the bridge.

**Path resolution rules** (per segment, left-to-right):
1. Look up an `FProperty` named `<segment>` on the current container's class.
2. If not found → `property_not_found at <path-so-far>`.
3. If found AND it's the last segment → coerce the value into this property and we're done.
4. If found AND not the last segment:
   - **FStructProperty** → use the struct's memory as the new container; continue with next segment.
   - **FObjectProperty** → dereference (`GetObjectPropertyValue`); if null, return `path_traversal_null at <path>`. Use the pointed-to UObject as the new container.
   - **Other types** → `path_traversal_invalid_type at <path>: cannot traverse through <FProperty class>` (e.g., can't path through a primitive int).

Plus all primitives + UE structs from v0.3.0 stay supported (no regression):
`bool`, `int8/16/32/64`, `uint8/16/32/64`, `float`, `double`, `FString`, `FText`, `FName`, `FVector`, `FVector2D`, `FRotator`, `FLinearColor`, `FColor`, `UEnum`, `TSoftObjectPtr<T>`.

### Out of scope (deferred to v0.4.x or later)

- **FInstancedStruct** (polymorphic struct) — needs `_type` discriminator design; its own brainstorm cycle.
- **Numeric-key TMaps** (`TMap<int32, V>`, etc.) — uncommon in editor properties; defer until usage shows the need.
- **TWeakObjectPtr / TLazyObjectPtr** — rare in editor APIs; defer.
- **Custom property setters** (`BlueprintSetter` UMETA) — use the generic FProperty write path; setter callbacks NOT invoked. Documented as a known limitation.
- **Recursion depth >8 levels** — explicit cap to prevent abuse / stack overflow.

## Cross-cutting design decisions

### Recursion + path tracking

`PropertyCoercion::SetProperty` and `EncodeProperty` get a new parameter: `const FString& CurrentPath`. As recursion descends, the path builds up:

```
. (root)
.RootComponent
.RootComponent.RelativeLocation
.RootComponent.RelativeLocation.X
.Materials[2]
.Materials[2].DiffuseColor
```

When a nested coercion fails, the error message includes the full path:

```
set_actor_property: unsupported_property_type at .Materials[2].DiffuseColor:
                    FInstancedStruct (deferred to v0.4.x)
```

Without path tracking, the LLM gets `unsupported_property_type` and has no idea *which* field of *which* element triggered it. Path-aware errors are crucial for the LLM to retry intelligently.

### Recursion depth limit

8 levels. Beyond → `recursion_depth_exceeded` error. Configurable via constant `kMaxCoercionDepth`. Realistically nobody hits 8 levels in editor properties; the cap exists to prevent stack-overflow abuse from malicious or buggy inputs.

### Encode (GET) / Decode (SET) symmetry

Every type that can be **set** can also be **read back**. `EncodeProperty` becomes the structural inverse of `SetProperty`, recursive in the same way.

This means after:
```json
set_actor_property name="Foo" property="MyArray" value=[1,2,3]
```

The response includes:
```json
{
  "old_value": [0, 0, 0],
  "new_value": [1, 2, 3]
}
```

…with full nested structure preserved through the JSON round-trip.

### Type-specific design notes

**USTRUCT handling:**
1. Get `UScriptStruct*` via `StructProp->Struct`.
2. Iterate properties via `TFieldIterator<FProperty>`.
3. For each property in the JSON object, find the matching `FProperty` by name.
4. Recursively call `PropertyCoercion::SetProperty` on that field within the struct's memory.
5. Fields not in the JSON are **left at their existing values** (partial updates allowed — the result includes the full new struct value).
6. Field names in the JSON that don't exist on the struct → error: `field_not_found at <path>: 'foo' on FCustomStruct`.

**TArray / TSet handling (parallel pattern):**
1. Get the inner property via `ArrayProp->Inner` / `SetProp->ElementProp`.
2. Use `FScriptArrayHelper` / `FScriptSetHelper` to manipulate the underlying container.
3. Resize / clear the container, then for each JSON array element, recursively coerce into the inner type and `Add`.
4. TSet automatically deduplicates by UE's set semantics (caller's responsibility to know).

**TMap handling (string-keyed only):**
1. Verify `MapProp->KeyProp` is `FStrProperty` or `FNameProperty`. If not → `unsupported_property_type at <path>: TMap with non-string key`.
2. Use `FScriptMapHelper` to manipulate the map.
3. Iterate JSON object keys.
4. For each key, recursively coerce the value via `MapProp->ValueProp`.
5. Insert into the map.

**FObjectProperty handling (hard pointer):**
1. Read JSON value as string (asset path).
2. Empty string → set to `nullptr`.
3. Non-empty → `LoadObject<UObject>(nullptr, *Path)`. If null → `asset_not_found at <path>: '/Game/Foo'`.
4. Type check: loaded object's class must be `IsChildOf(ObjectProp->PropertyClass)`. Otherwise → `wrong_object_class at <path>: expected UStaticMesh, got UMaterial`.
5. Set via `ObjectProp->SetObjectPropertyValue(PropAddr, LoadedObj)`.

### New stable error codes (joining the v0.2.0/v0.3.0 set)

| Code | Trigger |
|---|---|
| `field_not_found` | JSON object key doesn't exist on the target USTRUCT. |
| `wrong_object_class` | Loaded UObject's class doesn't match the property's required class. |
| `recursion_depth_exceeded` | Nested coercion exceeded `kMaxCoercionDepth` (8). |
| `path_traversal_null` | Path tried to traverse through a null FObjectProperty (e.g. uninitialized RootComponent). |
| `path_traversal_invalid_type` | Path tried to traverse through a non-traversable property type (primitives, arrays, maps). |
| `property_not_found` (existing, but enhanced) | Now reports the path-segment that didn't resolve. |
| `unsupported_property_type` (existing, but error message format extended) | Now includes the full property path. |

## Testing strategy

### Bridge unit tests (CI, no UE)

- `test_manifest_version_matches_bridge_server_version` enforces version sync; bumping both sides keeps it green.
- No new schema tests — `set_actor_property`'s wire schema is unchanged. The new capability is internal C++ coercion behavior, exercised only at runtime against UE.

### Live integration smoke test (manual, requires UE)

A new step "9. advanced property types round-trip" in `examples/smoke_test.py`:

```python
def t_advanced_props():
    # 1. Spawn a test actor
    spawn_resp = call("spawn_actor", {
        "class_path": "/Script/Engine.StaticMeshActor",
        "location": {"x": 0, "y": 0, "z": 0},
        "label": "AdvSmoke",
    })
    actor_name = assert_ok(spawn_resp, "spawn_actor.adv")["name"]

    # 2. Set a nested USTRUCT (RootComponent.RelativeLocation)
    set_loc = call("set_actor_property", {
        "name": actor_name,
        "property": "RootComponent.RelativeLocation",  # path traversal
        "value": {"x": 100, "y": 200, "z": 50},
    })
    set_loc_res = assert_ok(set_loc, "set_actor_property.struct")
    if set_loc_res["new_value"] != {"x":100, "y":200, "z":50}:
        raise SmokeFailure(f"USTRUCT round-trip failed: {set_loc_res}")

    # 3. Set a TArray<FName> (Tags)
    set_tags = call("set_actor_property", {
        "name": actor_name,
        "property": "Tags",
        "value": ["foo", "bar", "baz"],
    })
    set_tags_res = assert_ok(set_tags, "set_actor_property.array")
    if set_tags_res["new_value"] != ["foo", "bar", "baz"]:
        raise SmokeFailure(f"TArray round-trip failed: {set_tags_res}")

    # 4. Read-back via property path on a sub-component (nested USTRUCT GET)
    set_rot = call("set_actor_property", {
        "name": actor_name,
        "property": "RootComponent.RelativeRotation",
        "value": {"pitch": 0, "yaw": 90, "roll": 0},
    })
    assert_ok(set_rot, "set_actor_property.rotator")

    # 5. Cleanup
    delete = call("delete_actor", {"name": actor_name, "force": True})
    assert_ok(delete, "delete_actor.adv")
```

### Negative path tests (added to smoke):
- Set field that doesn't exist on a struct → expect `field_not_found at .X`.
- Set nested asset reference with wrong class → expect `wrong_object_class at .X`.
- Try to coerce a deeply nested structure exceeding `kMaxCoercionDepth` → expect `recursion_depth_exceeded`.

## Module dependencies

**No new module deps needed.** `FScriptArrayHelper`, `FScriptMapHelper`, `FScriptSetHelper`, `TFieldIterator`, `LoadObject` are all in `CoreUObject` which is already linked (was implicit in v0.3.0's `Engine` dep, made explicit in earlier follow-ups).

## Backward compatibility

Purely additive. Every v0.3.0 call continues to work unchanged. The new types are **new capability** — clients that don't use them are unaffected.

The `EncodeProperty` (read) path's signature changes (now takes `CurrentPath`), but `EncodeProperty` is internal-only — not exposed via MCP wire protocol.

The error messages from existing types now include path prefixes (e.g., `unsupported_property_type at .: ...` instead of `unsupported_property_type: ...`). This is technically a behavioral change but the path prefix is *additional* information — substring-matching clients keep working because the original code (`unsupported_property_type:`) still appears in the message.

## Forward compat

The path-tracking infrastructure becomes the foundation for any future advanced FProperty feature. v0.4.x extensions (FInstancedStruct, numeric-key maps, custom property setters) will plug into the same recursive framework without architectural changes.
