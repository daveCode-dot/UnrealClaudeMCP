# UnrealClaudeMCP v0.5.0 — Wire-protocol framing design

**Date:** 2026-05-08
**Status:** Approved — proceeding to implementation
**Target release:** v0.5.0
**Builds on:** v0.4.0 (advanced property types)

---

## Goal

Replace the **"one TCP `recv()` = one complete JSON message"** assumption in `MCPServer.cpp` and the bridge's recv loop with **length-prefixed framing**:

```
<8-byte big-endian uint64 length><N bytes of UTF-8 JSON body>
```

Removes the existing known limitation (`ARCHITECTURE.md:101`) and unblocks any future handler that needs to send/receive payloads larger than a single TCP `recv()` buffer.

**No new MCP handlers. Tool count stays at 19.** This is internal infrastructure.

## Why this matters

- Today, `get_viewport_screenshot` returns ~2 MB of base64 PNG. It works *most of the time* on localhost because Windows usually delivers the whole payload in one `recv()`. But it's not guaranteed — under congestion, system load, or kernel buffer fragmentation, a single `recv()` could return a partial message. The current bridge has a "data ends with `}`" heuristic that will fail on any JSON whose last char isn't `}` (impossible for valid JSON, but brittle reasoning).
- It blocks `create_texture_from_bytes` and any future handler that wants to ship raw bytes.
- It complicates ever batching responses or streaming output.

Length-prefixed framing fixes these in one ~200 LoC change.

### Out of scope (deferred)
- **Backward compatibility with v0.4.0 wire format.** Bridge and plugin must upgrade together. This is acceptable — the TCP wire is a *private internal contract* between bridge ↔ plugin (both shipped from this repo), not a public MCP-protocol surface. The MCP-over-stdio side facing Claude Code is unchanged; only the localhost TCP between the bridge and the UE plugin is affected.
- Streaming responses (still one request → one response).
- Compression / encryption (not needed on localhost; defer).

## Wire format

```
+--------+--------+--------+--------+--------+--------+--------+--------+--------+--------+
|     8-byte big-endian uint64 body length     |       N bytes of UTF-8 JSON body         |
+--------+--------+--------+--------+--------+--------+--------+--------+--------+--------+
```

- Length is the byte length of the JSON body (NOT including the 8-byte prefix itself).
- Length is encoded big-endian (network byte order); standard convention.
- Length max: 1 GB enforced (any prefix above that is rejected as `framing_error: length exceeds 1GB cap`).
- Empty body (length = 0) is invalid → `framing_error: zero-length body`.

## Design decisions

### Length size: 8 bytes vs 4 bytes

Chose 8 bytes (uint64) over 4 bytes (uint32, ~4 GB cap) because:
- The 4 extra bytes per message are negligible.
- Removes the ceiling forever; never need to revisit this format.
- Standard practice in modern protocols (gRPC, WebSocket, etc.).

### Length cap

1 GB per message. Anything larger → `framing_error`. Prevents memory exhaustion from malicious or buggy senders. Realistic UE responses cap out around 10–50 MB even for huge screenshots.

### Versioning

Bump `PROTOCOL_VERSION` (the bridge constant) and `transport.framing` field in `mcp_manifest.json`:

```json
"transport": {
  "type": "tcp",
  "host": "127.0.0.1",
  "port": 18888,
  "framing": "length-prefixed-uint64-be-json"
}
```

(Was `"framing": "json-per-message"`.)

### Failure mode for mismatched versions

If a v0.4.0 bridge connects to a v0.5.0 plugin (or vice versa), the framing mismatch produces an immediate `parse_error` on the receiver. **Loud, not silent.** Documented in PR description so anyone running a custom bridge knows to upgrade in lockstep.

### New stable error code

| Code | Trigger |
|---|---|
| `framing_error` | Length prefix unreadable, length exceeds 1 GB cap, length is zero, or body bytes don't arrive within socket timeout. |

## Backward compatibility

**Wire format break — explicit and documented.** The bridge and plugin must upgrade together. Mixed versions get an immediate parse error, not silent corruption. This is the right tradeoff for a private localhost protocol on a feature release.

PR description calls this out so anyone with their own custom bridge knows to upgrade.

The MCP-over-stdio interface (Claude Code ↔ bridge) is unchanged. Only the bridge ↔ plugin TCP layer changes.

## Module dependencies

No new module deps. `FArchive`, `ByteSwap`, `INTEL_ORDER64` all in `Core`.
