# UnrealClaudeMCP v0.3.0 — Build-a-Level design

**Date:** 2026-05-08
**Status:** Approved — proceeding to implementation plan
**Author:** Claude (with Najem)
**Target release:** v0.3.0
**Builds on:** v0.2.0 (texture pipeline, error-code prefixes, pyproject.toml, hero banner)

---

## Goal

Add 6 MCP handlers that let an LLM construct a non-trivial UE 5.7 scene end-to-end without falling back to `execute_unreal_python`. Tool count grows from 13 → 19.

The headline use case: "Spawn 10 cubes in a grid, configure each one's material, then animate them" should be doable with first-class tool calls instead of multi-line Python escape-hatch incantations.

## Scope

### In scope (this PR)

| Handler | Purpose |
|---|---|
| `find_assets` | Query the asset registry by class + path + name. Returns matching assets so the LLM can discover what's available to spawn. |
| `spawn_actor` | Create an actor at a location with optional rotation, label, and initial properties. |
| `set_actor_transform` | Move / rotate / scale an existing actor by name. |
| `delete_actor` | Remove an actor by name. |
| `set_actor_property` | Mutate any UPROPERTY on an actor. Supports primitives + common UE structs. |
| `add_component` | Attach a component to an existing actor at runtime, optionally socketed. |

### Out of scope (deferred — explicit list)

- **USTRUCT / TArray / TMap / TSet / FInstancedStruct property types** — v0.4.0 (asset-registry / advanced types deep dive)
- **Tag-based asset registry queries** — v0.4.0
- **Bulk operations** (`spawn_actors_batch`, `delete_actors_batch`) — speculative; YAGNI until usage shows the need
- **Component removal** (`remove_component`) — companion to `add_component`; can land as v0.3.1
- **Actor replication / network configuration** — domain-specific; defer
- **Sublevels and level streaming** — persistent level only in v0.3.0

## Cross-cutting decisions

### Identification scheme — Hybrid (label OR FName)

Write handlers (`set_actor_transform`, `delete_actor`, `set_actor_property`, `add_component`) accept either:

- A **label** (the visible name in World Outliner) — ergonomic, what users say
- An **FName** (e.g. `Cube_C_2`) — bulletproof, unique within the world

Disambiguation rule:
1. Try label match first. If exactly one actor matches → use it.
2. If multiple actors match the label → return `ambiguous_actor` error with all matching FNames listed. The LLM retries with a specific FName.
3. If zero actors match the label → try FName match.
4. If FName match fails → `actor_not_found` error.

**New stable error code:** `ambiguous_actor` (joins the existing codes from PR #5).

Rationale: matches the existing `focus_actor` precedent (label-or-FName), but adds explicit ambiguity detection for write operations where deleting the wrong actor is a real footgun.

### Error code prefix scheme

Same `<tool>: <error_code>: <human-readable detail>` pattern from PR #5. Each handler section below lists its stable error codes.

### Shared modules

Two cross-cutting concerns get extracted to shared modules instead of duplicated across 6 handlers:

- **`MCP/ActorIdentity.h/cpp`** (~120 LoC) — implements the hybrid label-or-FName lookup with ambiguity detection. Used by 5 handlers.
- **`MCP/PropertyCoercion.h/cpp`** (~300 LoC) — implements JSON → FProperty value coercion for the supported types. Used by 3 handlers (`spawn_actor.properties`, `set_actor_property`, `add_component.relative_transform`).

This is a small architectural step beyond v0.2.0's pure-leaf handler pattern. Each shared module has one clear purpose and a well-defined interface; the handlers stay leaves but lift cross-cutting concerns to where they can be tested once.

## Handler 1: `find_assets`

Query the asset registry by class + optional path + optional name substring. Returns matching assets with structured records.

### Params

| Field | Type | Required | Default | Notes |
|---|---|---|---|---|
| `class_path` | string | ✅ | — | UE class path, e.g. `/Script/Engine.StaticMesh`, `/Script/Engine.Blueprint`, `/Script/Engine.Texture2D`. |
| `path_under` | string | ❌ | `/Game/` | Recursive path filter. Must start with `/Game/` or `/Engine/`. |
| `name_contains` | string | ❌ | — | Case-insensitive substring filter on asset name. |
| `limit` | int | ❌ | `100` | Cap result count. Max `500` (silently capped). |

### Result

```json
{
  "ok": true,
  "matched": 42,
  "returned": 42,
  "assets": [
    { "name": "SM_Cube", "package_path": "/Engine/BasicShapes/Cube", "class": "StaticMesh" },
    { "name": "SM_Sphere", "package_path": "/Engine/BasicShapes/Sphere", "class": "StaticMesh" }
  ]
}
```

`matched` is the total before `limit`; `returned` is the actual count in the response.

### Errors

| Code | Trigger |
|---|---|
| `invalid_class_path` | `class_path` does not resolve to a known UClass. |
| `invalid_path_filter` | `path_under` doesn't start with `/Game/` or `/Engine/`. |

### Implementation approach

- Call `IAssetRegistry::Get().GetAssets(FARFilter)` with `ClassPaths` set from `class_path`, `PackagePaths` from `path_under`, `bRecursivePaths=true`.
- Filter results in-memory by `name_contains` (case-insensitive `FString::Contains`).
- Sort by name, apply `limit`.

## Handler 2: `spawn_actor`

Create an actor at a location with optional rotation, label override, and initial properties.

### Params

| Field | Type | Required | Default | Notes |
|---|---|---|---|---|
| `class_path` | string | ✅ | — | Actor class path. Built-in: `/Script/Engine.StaticMeshActor`. Blueprint: `/Game/Blueprints/BP_MyActor.BP_MyActor_C` (note the `_C` suffix for the generated class). |
| `location` | object | ❌ | `{x:0, y:0, z:0}` | World-space `{x: float, y: float, z: float}`. |
| `rotation` | object | ❌ | `{pitch:0, yaw:0, roll:0}` | Euler angles in degrees. |
| `label` | string | ❌ | UE auto-name | Visible name in World Outliner. |
| `properties` | object | ❌ | — | Map of `{ "PropertyName": value }` applied via `PropertyCoercion` immediately after spawn. |

### Result

```json
{
  "ok": true,
  "name": "BP_MyActor_C_4",
  "label": "BP_MyActor",
  "class": "BP_MyActor_C",
  "location": {"x": 0, "y": 0, "z": 0},
  "rotation": {"pitch": 0, "yaw": 0, "roll": 0}
}
```

### Errors

| Code | Trigger |
|---|---|
| `missing_required_field` | `class_path` missing/empty. |
| `invalid_class_path` | Class not found in asset registry. |
| `class_not_spawnable` | Class is abstract, doesn't derive from `AActor`, or has `bEditorOnly = true`. |
| `spawn_failed` | `UWorld::SpawnActor` returned null (UE refused for an internal reason). |
| `property_application_failed` | One of the `properties` failed to apply. The actor is still spawned; the error message lists which property failed. Caller can `delete_actor` for atomic-like cleanup. |

### Implementation approach

- Resolve `class_path` to `UClass*` via `LoadClass<AActor>(nullptr, *ClassPath)`.
- Validate it's spawnable (non-abstract, `IsChildOf<AActor>()`, not editor-only).
- Build `FActorSpawnParameters` with `Name` = `label` if provided.
- Call `GEditor->GetEditorWorldContext().World()->SpawnActor<AActor>(Class, Location, Rotation, SpawnParams)`.
- If `properties` provided, iterate and call `PropertyCoercion::SetProperty(Actor, Name, Value)` for each.
- If `label` was provided, call `Actor->SetActorLabel(Label)` post-spawn (the `Name` field affects FName, not label).

## Handler 3: `set_actor_transform`

Move / rotate / scale an actor by name.

### Params

| Field | Type | Required | Default | Notes |
|---|---|---|---|---|
| `name` | string | ✅ | — | Label or FName (hybrid identification). |
| `location` | object | ❌ | unchanged | `{x, y, z}`. |
| `rotation` | object | ❌ | unchanged | `{pitch, yaw, roll}`. |
| `scale` | object | ❌ | unchanged | `{x, y, z}` — multiplier. |
| `relative` | bool | ❌ | `false` | When `true`, deltas are added to current values instead of replacing them. |

### Result

```json
{
  "ok": true,
  "name": "Cube_C_2",
  "applied": {
    "location": {"x": 100, "y": 0, "z": 50},
    "rotation": {"pitch": 0, "yaw": 45, "roll": 0}
  }
}
```

`applied` only contains the fields actually changed.

### Errors

| Code | Trigger |
|---|---|
| `actor_not_found` | No actor matches the name. |
| `ambiguous_actor` | Label matches multiple actors. Error message lists candidate FNames. |
| `no_changes_specified` | None of `location` / `rotation` / `scale` provided. |

### Implementation approach

- Resolve actor via `ActorIdentity::Resolve(Name, OutAmbiguityList)`.
- Read current transform, compose new transform based on which fields are present and whether `relative` is set.
- Call `Actor->SetActorTransform(NewTransform)`.

## Handler 4: `delete_actor`

Remove an actor by name.

### Params

| Field | Type | Required | Default | Notes |
|---|---|---|---|---|
| `name` | string | ✅ | — | Label or FName. |
| `force` | bool | ❌ | `false` | When `false`, refuses to delete if the actor has attached children → `has_children` error. When `true`, deletes anyway (children become detached, mirroring UE's delete-with-children behavior). |

### Result

```json
{ "ok": true, "name": "Cube_C_2", "deleted": true }
```

### Errors

| Code | Trigger |
|---|---|
| `actor_not_found` | No actor matches. |
| `ambiguous_actor` | Label matches multiple actors. |
| `has_children` | `force=false` and actor has attached children. |

### Implementation approach

- Resolve actor via `ActorIdentity::Resolve`.
- Check attached children count via `Actor->GetAttachedActors(OutChildren)`.
- If `force=false` and children > 0 → return `has_children` error.
- Call `World->DestroyActor(Actor)`.

## Handler 5: `set_actor_property`

Mutate a UPROPERTY on any actor.

### Params

| Field | Type | Required | Default | Notes |
|---|---|---|---|---|
| `name` | string | ✅ | — | Label or FName. |
| `property` | string | ✅ | — | UPROPERTY name (case-sensitive, matches C++ exactly). |
| `value` | any | ✅ | — | JSON value coerced based on the FProperty type. |

### Supported types in v0.3.0

This is the explicit scope wall. Anything not on this list returns `unsupported_property_type`.

| FProperty C++ type | JSON value shape |
|---|---|
| `bool` | JSON bool |
| `int8`, `int16`, `int32`, `int64` | JSON number (range-checked) |
| `uint8`, `uint16`, `uint32`, `uint64` | JSON number (range-checked) |
| `float`, `double` | JSON number |
| `FString` | JSON string |
| `FText` | JSON string |
| `FName` | JSON string |
| `FVector` | `{x, y, z}` |
| `FVector2D` | `{x, y}` |
| `FRotator` | `{pitch, yaw, roll}` |
| `FLinearColor` | `{r, g, b, a}` (0–1 floats) |
| `FColor` | `{r, g, b, a}` (0–255 ints) |
| Enum (`UEnum`-decorated) | JSON string (the enum value name) |
| `TSoftObjectPtr<T>` | JSON string (asset path) |

### Result

```json
{
  "ok": true,
  "name": "PointLight_2",
  "property": "Intensity",
  "old_value": 5000.0,
  "new_value": 12000.0
}
```

`old_value` and `new_value` are JSON-encoded using the same coercion (so they round-trip).

### Errors

| Code | Trigger |
|---|---|
| `actor_not_found` | No actor matches. |
| `ambiguous_actor` | Label matches multiple actors. |
| `property_not_found` | UPROPERTY with that name doesn't exist on the actor's class. |
| `unsupported_property_type` | Property exists but its FProperty class isn't in the v0.3.0 supported list. Error message lists the FProperty class name (e.g. `"StructProperty(MyCustomStruct)"`) so the LLM can fall back to `execute_unreal_python`. |
| `value_coercion_failed` | Value couldn't be coerced (e.g., string given for an int, range overflow). |

### Implementation approach

- Resolve actor via `ActorIdentity::Resolve`.
- Find the `FProperty*` via `Actor->GetClass()->FindPropertyByName(FName(*Property))`.
- Read current value (encode via `PropertyCoercion::EncodeJson`).
- Coerce JSON value to FProperty native via `PropertyCoercion::SetProperty(Actor, Property, Value)`.
- Call `Actor->PostEditChangeProperty(...)` to fire UE's edit cascade.
- Return old + new values in JSON form.

## Handler 6: `add_component`

Attach a component to an existing actor at runtime.

### Params

| Field | Type | Required | Default | Notes |
|---|---|---|---|---|
| `actor_name` | string | ✅ | — | Label or FName of the host actor. |
| `class_path` | string | ✅ | — | Component class path, e.g. `/Script/Engine.StaticMeshComponent`. |
| `component_name` | string | ❌ | UE auto-name | FName for the new component. |
| `attach_to` | string | ❌ | root component | Name of an existing component on the actor to attach to. |
| `socket` | string | ❌ | — | Socket name on the parent component. |
| `relative_transform` | object | ❌ | identity | `{location, rotation, scale}` relative to the parent. Same shape as `set_actor_transform`. |

### Result

```json
{
  "ok": true,
  "actor": "Cube_C_2",
  "component": "StaticMeshComponent_3",
  "class": "StaticMeshComponent",
  "attached_to": "DefaultSceneRoot"
}
```

### Errors

| Code | Trigger |
|---|---|
| `actor_not_found` | No actor matches. |
| `ambiguous_actor` | Label matches multiple actors. |
| `invalid_component_class` | Class not found, abstract, or not a `USceneComponent`/`UActorComponent` subclass. |
| `attach_target_not_found` | `attach_to` specified but no component with that name on the actor. |
| `socket_not_found` | `socket` specified but the parent component doesn't have that socket. |

### Implementation approach

- Resolve actor.
- Resolve component class via `LoadClass<UActorComponent>(nullptr, *ClassPath)`.
- Validate it's instantiable and a `UActorComponent` subclass.
- Construct via `NewObject<UActorComponent>(Actor, Class, FName(*ComponentName))`.
- Register: `Component->RegisterComponent()`.
- If it's a `USceneComponent`: attach via `AttachToComponent(Parent, FAttachmentTransformRules::KeepRelativeTransform, FName(*Socket))`.
- Apply `relative_transform` if provided.
- Mark actor as edited via `Actor->Modify()` and `Actor->RerunConstructionScripts()` if needed.

## Testing strategy

### Bridge unit tests (CI, no UE)

- 6 new schema tests in `tests/test_bridge.py`, one per handler, asserting required-field shape and type expectations.
- The parametrized `test_every_tool_routes_through_tools_call` automatically extends to 19 tools.
- `test_manifest_tool_count_matches_bridge` count bumped 13 → 19.
- pytest count grows 59 → 65.

### Live integration smoke test (manual, requires UE)

A new "build-a-level round-trip" step in `examples/smoke_test.py`:

```
1. find_assets class=/Script/Engine.StaticMesh path=/Engine/BasicShapes/
   → expect Cube/Sphere/Cylinder/Cone/Plane in result
2. spawn_actor class=/Script/Engine.StaticMeshActor location={0,0,0} label="SmokeCube1"
3. spawn_actor class=/Script/Engine.StaticMeshActor location={200,0,0} label="SmokeCube2"
4. set_actor_transform name="SmokeCube2" location={200,200,50} rotation={0,45,0}
5. set_actor_property name="SmokeCube1" property="bHidden" value=false
6. add_component actor_name="SmokeCube1" class=/Script/Engine.PointLightComponent
7. set_actor_property name="<PointLightComponent_X>" property="Intensity" value=15000
   (uses the FName returned from add_component)
8. delete_actor name="SmokeCube1" force=true
9. delete_actor name="SmokeCube2"
10. Verify world is back to baseline (no SmokeCube* actors remain)
```

### Negative path tests

Live (smoke):
- Ambiguous label: spawn 2 actors with the same label, then `set_actor_transform` by label → expect `ambiguous_actor` error with both FNames listed.
- Invalid class path → `invalid_class_path`.
- Unsupported property type: try to set a USTRUCT-typed property → `unsupported_property_type`.
- Attach to nonexistent component → `attach_target_not_found`.

Bridge (CI):
- Missing required fields → schema test catches at param-validation level.
- Unknown enum-style values are not in this PR (no enum params on these handlers — strings are class paths or property names, not bounded enums).

## Forward compatibility note

This is a purely additive release. No existing v0.2.0 tool changes signature, returns shape, or error contract. Tool count goes 13 → 19. Anyone who built scripts on top of v0.2.0 keeps working unchanged.

The shared modules (`ActorIdentity` + `PropertyCoercion`) are internal implementation detail — they're not exposed via MCP and don't introduce any wire-protocol changes.

The supported-property-types list is explicitly documented in `set_actor_property`'s spec. Future versions may extend the list (additive); they will not remove from it (would be a breaking change).
